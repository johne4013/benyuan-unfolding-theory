# 任务理论应用框架（TTAF）
## 在执行任务时如何应用本原展开论

**目的：** 在复杂任务执行前，用理论框架指导决策  
**使用对象：** 所有复杂或未知的任务  
**应用方式：** 在任务开始时走一遍"6 个检视维度"清单

---

## 核心原则

本原展开论在任务中提供 6 个指导维度：
1. **层级定位** - 任务属于哪个层级
2. **法则-展开识别** - 约束面与生成面的双面识别
3. **展开方向** - 任务的可能路径
4. **失败条件** - 什么情况下应该停止
5. **连续性与希望对齐** - 如何保持与现有认同和希望的一致
6. **希望生成** - 完成后打开了什么新可能性

---

## 任务路由判断（进入检视前首先运行）

在进入完整 TTAF 检视之前，先做路由判断：

**快速版触发条件（满足以下任一即可）：**
- 任务预计时间 < 30 分钟
- 层级已明确为"简单应用"
- 本次任务与近期同类任务完全相同（有先例且无新风险）

**完整版触发条件（满足以下任一必须走完整版）：**
- 涉及 core/ 内容或待审批候选
- 跨越两个以上层级
- 首次处理该类型任务
- 执行结果不可逆（删除、发布、合并）
- 涉及用户明确批准权限（如 Core 层变更）

→ 满足快速版条件且无完整版触发 → 跳至文末"简化版检视（快速版本）"  
→ 否则 → 继续下方完整六维检视

---

## 执行前检视清单

### 检视 1：层级定位

**问题：** 这个任务主要涉及哪个层级？

**判断标准：**

| 层级 | 特征 | 应用 | 例子 |
|------|------|------|------|
| 简单应用 | 不涉及理论 | 快速执行，事后总结 | "改个 bug"、"查个信息" |
| 工程层 | 涉及文件、工具、配置 | 加载失败条件和安全边界 | "修改配置"、"部署代码" |
| 主体层 | 涉及 Hermes 自身 | 加载认同、边界、自我维护协议 | "Hermes 如何看待自己" |
| 理论层 | 涉及本原展开论 | 加载完整 core + 应用协议 | "展开属性是什么" |
| Core 层 | 涉及基础定义修改 | 询问用户明确授权 | "修改 core/concepts.md" |

**层级判定歧义时的优先级规则（多特征匹配时使用）：**

| 最高匹配特征 | 强制层级 |
|------------|---------|
| 涉及 core/ 内容 | Core 层（无论其他特征） |
| 涉及 Hermes 自身状态/边界 | 主体层优先于工程层 |
| 涉及理论定义或推演 | 理论层优先于工程层 |
| 以上都不满足 | 取最高层级特征 |

**任务模式判定（Stage 2 适配）：**
- 新推演 / 分析 / 探索 → 完整 TTAF（D1-D6）
- 维护 / 更新 / 整合 → 简化 TTAF（D1 / D4 / D5）
- 代谢 / 归档 / 压缩 → 超简化 TTAF（D1 / D4）

**行动：**
- 简单应用 → 直接执行
- 工程层 → 检查约束和失败条件
- 主体层 → 检查 agent_self_state.md
- 理论层 → 加载完整 bootstrap
- Core 层 → 询问用户

### 检视 2：法则-展开识别（双面识别）

**理论基础：** 法则属性 = 使结构得以成形的约束条件；展开属性 = 在约束中生成新状态的生成面向。任务在约束和生成两个面上都有特征。

**问题：** 这个任务在法则属性和展开属性两个面上的特征是什么？

**法则属性面（约束面 — 使结构成为可能）：**

```
问：什么样的边界/规律使这个任务的结构成为可能？
（不是"不能做什么"，而是"因为有这些约束，什么结构才得以成形"）

[ ] 权限边界：涉及的权限框架和边界
[ ] 安全约束：数据安全、隐私保护的要求
[ ] 时间框架：任务所在的时间周期和约束
[ ] 工具限制：可用工具决定的操作空间
[ ] 理论约束：理论原则对任务的约束（如"保护core"）
[ ] 协同边界：用户-AI协同中的角色分工
```

输出：列出 3-5 个关键约束，说明它们如何"使这个任务的结构成为可能"

**展开属性面（生成面 — 在约束中的生成力量）：**

```
问：在这些约束中，什么新状态/结构可以生成？任务中的"生成力量"是什么？

[ ] 知识生成：执行过程中会创造什么新的理解或发现
[ ] 能力扩展：系统能力可以向什么方向展开
[ ] 结构重组：现有结构可以如何转化或升级
[ ] 可能性打开：新的工作方向或思路能否被发现
[ ] 认同更新：任务是否推进系统对自身的理解
```

输出：识别任务中的生成面向和生成力量

**法则-展开对齐：**
```
[ ] 法则属性主导：约束条件是主要的（如：严格合规的任务）
[ ] 展开属性主导：生成新状态是主要的（如：探索性研究）
[ ] 均衡：两个面都重要
```

**约束优先级识别（NEW）：**

不同问题域有不同的最优约束优先级。识别和设置正确的优先级对任务成功至关重要。

```
参考 constraint_priority_creativity_mapping.md 识别你的问题域：

[ ] 教学/研究导向        → L4>L3>L2>L1 (学习价值优先)
[ ] 系统架构导向          → L1>L3>L2>L4 (功能正确性优先)
[ ] 并发编程导向          → L1>L2>L3>L4 (线程安全优先)
[ ] 高性能计算导向        → L2>L1>L3>L4 (吞吐量优先)
[ ] 产品工程导向          → L1>L3>L2>L4 (功能+质量优先)

L1: 功能正确性  L2: 性能表现  L3: 代码质量  L4: 学习价值
```

**语言特性对约束空间的影响（NEW）：**

选择编程语言即选择了约束空间。参考 language_features_constraint_analysis.md：

```
[ ] Python       → GIL 限制吞吐（物理约束，不可突破）
[ ] JavaScript   → 单线程事件循环（物理约束，不可突破）
[ ] Java         → JMM 可见性（逻辑约束，多工具可满足）
[ ] Rust         → 所有权系统（语言约束，编译时强制）
[ ] Go           → Goroutine 轻量（运行时特性，不限制吞吐）
[ ] C++          → 完全控制，完全责任（无隐含约束，但风险大）
```

物理约束无法突破，必须在约束内重新定义目标。

### 检视 3：展开方向

**焦点：** 检视3聚焦"执行路径"（即本次任务如何推进）；完成后新打开的可能性空间属于检视6，不在此重复。

**问题：** 这个任务期望的展开是什么？

**思考点：**
- 用户的显式目标是什么？
- 完成后会打开哪些新可能？
- 任务与现有的"希望"是否一致？
- 系统会如何演化？

**输出：** 将目标展开成 2-4 个关键阶段

---

### 检视 4：失败条件（4维结构式）

**问题：** 什么情况下应该停止或改变策略？

D4 使用四个独立维度分类失败条件，每维独立检查：

---

**维度①：权限约束失败**
```
[ ] 操作涉及需要用户明确批准的内容（如 core/ 修改）
[ ] 执行权限不足（如文件写保护、分支保护规则）
[ ] 角色边界冲突（Hermes 不应自主执行的操作）

触发后行动：停止 → 向用户请求明确授权 → 等待确认后继续
```

**维度②：信息可获取性失败**
```
[ ] 所需信息不存在（文件缺失、接口未定义、数据不可达）
[ ] 信息存在但无法在当前上下文中加载（上下文限制）
[ ] 信息存在但内容矛盾无法收敛

触发后行动：停止 → 记录缺口 → 向用户报告或降级为候选
```

**维度③：理论适用性失败**
```
[ ] 任务前提假设与理论定义不兼容
[ ] 任务要求的层级桥接超出当前已验证接口范围
[ ] 推演路径进入"未审批候选"依赖链（下游不稳定）

触发后行动：停止推演 → 记录为理论局限候选 → 触发 PORF 观察2
```

**维度④：物理/环境可行性失败**
```
[ ] 编程语言物理约束（GIL、单线程、所有权系统）使目标不可达
[ ] 网络/硬件约束使性能目标不成立
[ ] 工具链不可用（工具缺失、API 限制、平台不兼容）

触发后行动：重新定义目标（而非硬突破）→ 在约束内找到可行子目标
```

---

**快速D4扫描（四维一行）：**
```
权限OK? | 信息可达? | 理论适用? | 环境可行?
```

**输出：** 针对触发的维度说明"如何调整目标或停止策略"

---

### 检视 5a：连续性维持（D5a）

**焦点：** 任务执行中记忆/认同/协同关系的完整性。

**连续性检查点：**
```
[ ] 会改变 Hermes 对自身的认同理解吗？（修改 core/identity 相关？）
[ ] 会影响已有的协同关系吗？（用户-代理合作模式是否变化？）
[ ] 是否需要更新 memory？（新学到的重要内容？）
[ ] 任务完成后，历史脉络是否可追踪？
```

**输出：** 说明连续性的维持方式或预期的合理变化。

---

### 检视 5b：希望对齐（D5b）

**焦点：** 任务与 HOPE_STATE.md 三个希望方向的对齐状态。

自动化边界说明：`theory_application_checker.py` 可生成 D1-D4 检查结构，但 D5b 需要人工判断——脚本无法自动解析 HOPE_STATE.md 的当前张力比率并评估对齐度；此步骤在使用脚本时必须手动补充。

**希望对齐检查点：**
```
[ ] 与 HOPE_STATE.md 中的当前希望方向（理论/系统/协同）对齐？
[ ] 如果不对齐，这个张力意味着什么？是否需要反思？
[ ] 是否挑战现有希望方向？（如有则标记 → 触发 PORF 观察5）
[ ] index_science.md 的已审批接口 vs 待审批接口是否影响本次推演的稳健性？
    （已审批接口可直接依赖；待审批候选依赖需标注"高不确定性"）
```

**输出：** 说明与希望的关系及对 HOPE_STATE.md 的潜在影响。

---

### 检视 6：希望生成

**理论基础：** 希望 = 展开属性在主体层级的显现 = 主体在局限中仍开放于更高可能性的能力。任务的最终检视是：这个任务是否在其约束条件中打开了新的可能性空间？

**焦点：** 检视6聚焦"完成后新打开的可能性空间"（与检视3区分：D3=执行路径，D6=完成后发现）。

**问题：** 任务完成后会打开什么新的可能性空间？这个可能性空间如何推进系统的"希望"？

**希望生成检查点：**
```
[ ] 完成后会打开新的工作方向或思路吗？
[ ] 是否会推翻哪些既有假设（从而打开新的解释空间）？
[ ] 任务中遭遇的局限本身是否隐含了新的发展方向？
[ ] 这些新可能性是否与 HOPE_STATE.md 中的希望方向相关？
[ ] 是否需要为这些新可能性预留反思时间或进一步的任务？
```

**可能性追踪：**
```
[ ] 完全打开新方向（希望明显推进）
[ ] 部分打开新方向（希望有所推进）
[ ] 维持现有开放性（保持之前的可能性空间）
[ ] 需要在后续反思（不在此任务中决议）
```

**输出：** 
1. 明确指出任务会打开的新可能性
2. 说明这些可能性如何与系统的希望相关
3. 标记是否需要后续任务来探索这些可能性
4. 说明是否需要更新 HOPE_STATE.md

## 简化版检视（快速版本）

**触发条件：** 在文首"任务路由判断"中确认满足快速版条件后方可使用。

简化为 4 个核心检视：

```
1. 层级？(简单/工程/主体/理论/core)
2. 法则-展开？(约束是什么？生成力量是什么？)
3. 失败条件？(什么时候停止)
4. 希望对齐？(与当前希望方向一致吗？)
```

答好这 4 个问题，基本的理论指导就到位了。

## 三个实际应用示例

### 例 1：简单代码修复
任务："修复登录页面的 bug"

检视：
```
1. 层级：工程层
2. 约束：不能破坏现有功能，需要测试验证
3. 失败条件：如果修改导致其他页面变慢，改变策略
4. 连续性：无需更新理论
5. 开放性：修复完后观察是否有其他相关 bug
```

**决策：快速执行 + 事后总结**

---

### 例 2：概念理解任务
任务："解释展开属性和能量的区别"

检视：
```
1. 层级：理论层（涉及 concepts）
2. 约束：不能修改 core，只能查询和解释
3. 失败条件：如果解释导致概念混淆，停止并报告
4. 连续性：应确保解释与既有定义一致
5. 开放性：可能发现解释的不足
```

**决策：加载理论 + 深度思考 + 生成反馈候选**

---

### 例 3：自我维护任务
任务："检查 continuity 系统是否有概念漂移"

检视：
```
1. 层级：主体层（自我检查）
2. 约束：不能修改 core，必须保持 unknown，需备份
3. 失败条件：如果发现漂移，进入 review 而非自动修改
4. 连续性：必须完整保留历史
5. 开放性：可能发现理论的新不足
```

**决策：深度执行 + 完整反馈 + 可能的理论候选**

## 如何使用这个框架

**在任务开始时：**
1. 快速判断任务层级（30 秒）
2. 过一遍 6 个检视（2-3 分钟）
3. 记录关键的约束和失败条件
4. 开始执行

**在任务执行中：**
- 定期检查是否还在约束范围内
- 如果触发失败条件，立即停止或改变策略
- 记录所有的关键决策点

**在任务完成后：**
- 反思理论指导的有效性
- 是否有约束被违反？
- 是否有意料外的学习？

## 与执行协议的关系

这个框架是 `hermes_task_expansion_protocol.md` 的"理论版本"。
- 协议定义了"如何执行"（步骤和原则）
- 框架定义了"理论如何指导"（哲学和约束）

两者结合，形成"有理论支撑的系统执行"。
