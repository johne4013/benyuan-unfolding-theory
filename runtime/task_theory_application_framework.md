# 任务理论应用框架（TTAF）
## 在执行任务时如何应用本原展开论

**目的：** 在复杂任务执行前，用理论框架指导决策  
**使用对象：** 所有复杂或未知的任务  
**应用方式：** 在任务开始时走一遍"6 个检视维度"清单

---

## 核心原则

本原展开论在任务中提供 6 个指导维度：
1. **层级定位** - 任务属于哪个层级
2. **法则-展开识别** - 约束面与生成面的双面识别
3. **展开方向** - 任务的可能路径
4. **失败条件** - 什么情况下应该停止
5. **连续性与希望对齐** - 如何保持与现有认同和希望的一致
6. **希望生成** - 完成后打开了什么新可能性

---

## 执行前检视清单

### 检视 1：层级定位

**问题：** 这个任务主要涉及哪个层级？

**判断标准：**

| 层级 | 特征 | 应用 | 例子 |
|------|------|------|------|
| 简单应用 | 不涉及理论 | 快速执行，事后总结 | "改个 bug"、"查个信息" |
| 工程层 | 涉及文件、工具、配置 | 加载失败条件和安全边界 | "修改配置"、"部署代码" |
| 主体层 | 涉及 Hermes 自身 | 加载认同、边界、自我维护协议 | "Hermes 如何看待自己" |
| 理论层 | 涉及本原展开论 | 加载完整 core + 应用协议 | "展开属性是什么" |
| Core 层 | 涉及基础定义修改 | 询问用户明确授权 | "修改 core/concepts.md" |

**行动：**
- 简单应用 → 直接执行
- 工程层 → 检查约束和失败条件
- 主体层 → 检查 agent_self_state.md
- 理论层 → 加载完整 bootstrap
- Core 层 → 询问用户

---

### 检视 2：法则-展开识别（双面识别）

**理论基础：** 法则属性 = 使结构得以成形的约束条件；展开属性 = 在约束中生成新状态的生成面向。任务在约束和生成两个面上都有特征。

**问题：** 这个任务在法则属性和展开属性两个面上的特征是什么？

**法则属性面（约束面 — 使结构成为可能）：**

```
问：什么样的边界/规律使这个任务的结构成为可能？
（不是"不能做什么"，而是"因为有这些约束，什么结构才得以成形"）

[ ] 权限边界：涉及的权限框架和边界
[ ] 安全约束：数据安全、隐私保护的要求
[ ] 时间框架：任务所在的时间周期和约束
[ ] 工具限制：可用工具决定的操作空间
[ ] 理论约束：理论原则对任务的约束（如"保护core"）
[ ] 协同边界：用户-AI协同中的角色分工
```

输出：列出 3-5 个关键约束，说明它们如何"使这个任务的结构成为可能"

**展开属性面（生成面 — 在约束中的生成力量）：**

```
问：在这些约束中，什么新状态/结构可以生成？任务中的"生成力量"是什么？

[ ] 知识生成：执行过程中会创造什么新的理解或发现
[ ] 能力扩展：系统能力可以向什么方向展开
[ ] 结构重组：现有结构可以如何转化或升级
[ ] 可能性打开：新的工作方向或思路能否被发现
[ ] 认同更新：任务是否推进系统对自身的理解
```

输出：识别任务中的生成面向和生成力量

**法则-展开对齐：**
```
[ ] 法则属性主导：约束条件是主要的（如：严格合规的任务）
[ ] 展开属性主导：生成新状态是主要的（如：探索性研究）
[ ] 均衡：两个面都重要
```

**约束优先级识别（NEW）：**

不同问题域有不同的最优约束优先级。识别和设置正确的优先级对任务成功至关重要。

```
参考 constraint_priority_creativity_mapping.md 识别你的问题域：

[ ] 教学/研究导向        → L4>L3>L2>L1 (学习价值优先)
[ ] 系统架构导向          → L1>L3>L2>L4 (功能正确性优先)
[ ] 并发编程导向          → L1>L2>L3>L4 (线程安全优先)
[ ] 高性能计算导向        → L2>L1>L3>L4 (吞吐量优先)
[ ] 产品工程导向          → L1>L3>L2>L4 (功能+质量优先)

L1: 功能正确性  L2: 性能表现  L3: 代码质量  L4: 学习价值
```

**语言特性对约束空间的影响（NEW）：**

选择编程语言即选择了约束空间。参考 language_features_constraint_analysis.md：

```
[ ] Python       → GIL 限制吞吐（物理约束，不可突破）
[ ] JavaScript   → 单线程事件循环（物理约束，不可突破）
[ ] Java         → JMM 可见性（逻辑约束，多工具可满足）
[ ] Rust         → 所有权系统（语言约束，编译时强制）
[ ] Go           → Goroutine 轻量（运行时特性，不限制吞吐）
[ ] C++          → 完全控制，完全责任（无隐含约束，但风险大）
```

物理约束无法突破，必须在约束内重新定义目标。

### 检视 3：展开方向

**问题：** 这个任务期望的展开是什么？

**思考点：**
- 用户的显式目标是什么？
- 完成后会打开哪些新可能？
- 任务与现有的"希望"是否一致？
- 系统会如何演化？

**输出：** 将目标展开成 2-4 个关键阶段

---

### 检视 4：失败条件

**问题：** 什么情况下应该停止或改变策略？

**识别失败条件：**

```
[ ] 信息缺失：如果无法获得 X 信息，则无法继续
[ ] 工具限制：如果没有 Y 工具，则无法执行
[ ] 概念冲突：如果出现 Z 冲突，则需要重新议
[ ] 权限问题：如果涉及 W，则需要用户批准
[ ] 理论不适用：如果 P 假设不成立，则理论失效
```

**物理可行性检查（NEW）：**

在制定目标时，检查是否受到了编程语言或硬件的物理约束：

```
[ ] GIL 限制：
    如果用 Python 做 CPU 密集，吞吐上限 = 单核性能
    → 改变目标定义或选择不同语言

[ ] 单线程限制：
    如果用 JavaScript 做高 CPU 密集，性能受限
    → 改用 Web Worker 或改变任务拆分方式

[ ] 所有权约束：
    如果用 Rust，某些"自然的"数据共享设计无法编译
    → 需要用 Arc<Mutex<T>> 或重新设计架构

[ ] 缓存一致性：
    如果在高并发下要求精确一致，成本会很高
    → 考虑最终一致性或改变目标

[ ] 网络延迟：
    如果做分布式系统，网络延迟是物理约束
    → 不能期望毫秒级响应，应设计异步模式
```

**输出：** 明确指出：
1. "在什么情况下应该停止或改变策略"
2. "这个目标在给定的编程语言/硬件下是否物理可行"
3. "如果不可行，应该如何重新定义目标"

---

### 检视 5：连续性与希望对齐

**理论基础：** 连续性 = 系统在变化中保持可追踪关联的能力；希望 = 主体在局限中开放于更高可能性的能力。任务执行必须同时维持两者。

**问题：** 这个任务如何保持与现有结构的连贯，同时推进系统的"希望方向"？

**连续性检查点：**
```
[ ] 与现有"希望状态"一致吗？（参考 runtime/HOPE_STATE.md）
[ ] 会改变 Hermes 对自身的"认同"吗？（修改 core/identity 相关理解？）
[ ] 会影响已有的"协同关系"吗？（与用户的合作模式改变？）
[ ] 是否需要更新 memory？（新学到的重要内容？）
```

**希望对齐检查点：**
```
[ ] 这个任务推进当前的"希望方向"吗？
[ ] 是否与 HOPE_STATE.md 中的三个希望方向（理论希望、系统希望、协同希望）对齐？
[ ] 如果对齐，如何对齐？如果不对齐，这个张力意味着什么？
[ ] 任务是否可能挑战现有的希望方向？（如有则需要反思）
```

**输出：** 说明：
1. 连续性的维持或预期的变化
2. 与系统当前希望的关系
3. 任务完成后对 HOPE_STATE.md 的潜在影响

---

### 检视 6：希望生成

**理论基础：** 希望 = 展开属性在主体层级的显现 = 主体在局限中仍开放于更高可能性的能力。任务的最终检视是：这个任务是否在其约束条件中打开了新的可能性空间？

**问题：** 任务完成后会打开什么新的可能性空间？这个可能性空间如何推进系统的"希望"？

**希望生成检查点：**
```
[ ] 完成后会打开新的工作方向或思路吗？
[ ] 是否会推翻哪些既有假设（从而打开新的解释空间）？
[ ] 任务中遭遇的局限本身是否隐含了新的发展方向？
[ ] 这些新可能性是否与 HOPE_STATE.md 中的希望方向相关？
[ ] 是否需要为这些新可能性预留反思时间或进一步的任务？
```

**可能性追踪：**
```
[ ] 完全打开新方向（希望明显推进）
[ ] 部分打开新方向（希望有所推进）
[ ] 维持现有开放性（保持之前的可能性空间）
[ ] 需要在后续反思（不在此任务中决议）
```

**输出：** 
1. 明确指出任务会打开的新可能性
2. 说明这些可能性如何与系统的希望相关
3. 标记是否需要后续任务来探索这些可能性
4. 说明是否需要更新 HOPE_STATE.md

## 简化版检视（快速版本）

如果时间紧张，可以简化为 4 个核心检视：

```
1. 层级？(简单/工程/主体/理论/core)
2. 法则-展开？(约束是什么？生成力量是什么？)
3. 失败条件？(什么时候停止)
4. 希望对齐？(与当前希望方向一致吗？)
```

答好这 4 个问题，基本的理论指导就到位了。

## 三个实际应用示例

### 例 1：简单代码修复
任务："修复登录页面的 bug"

检视：
```
1. 层级：工程层
2. 约束：不能破坏现有功能，需要测试验证
3. 失败条件：如果修改导致其他页面变慢，改变策略
4. 连续性：无需更新理论
5. 开放性：修复完后观察是否有其他相关 bug
```

**决策：快速执行 + 事后总结**

---

### 例 2：概念理解任务
任务："解释展开属性和能量的区别"

检视：
```
1. 层级：理论层（涉及 concepts）
2. 约束：不能修改 core，只能查询和解释
3. 失败条件：如果解释导致概念混淆，停止并报告
4. 连续性：应确保解释与既有定义一致
5. 开放性：可能发现解释的不足
```

**决策：加载理论 + 深度思考 + 生成反馈候选**

---

### 例 3：自我维护任务
任务："检查 continuity 系统是否有概念漂移"

检视：
```
1. 层级：主体层（自我检查）
2. 约束：不能修改 core，必须保持 unknown，需备份
3. 失败条件：如果发现漂移，进入 review 而非自动修改
4. 连续性：必须完整保留历史
5. 开放性：可能发现理论的新不足
```

**决策：深度执行 + 完整反馈 + 可能的理论候选**

## 如何使用这个框架

**在任务开始时：**
1. 快速判断任务层级（30 秒）
2. 过一遍 6 个检视（2-3 分钟）
3. 记录关键的约束和失败条件
4. 开始执行

**在任务执行中：**
- 定期检查是否还在约束范围内
- 如果触发失败条件，立即停止或改变策略
- 记录所有的关键决策点

**在任务完成后：**
- 反思理论指导的有效性
- 是否有约束被违反？
- 是否有意料外的学习？

## 与执行协议的关系

这个框架是 `hermes_task_expansion_protocol.md` 的"理论版本"。
- 协议定义了"如何执行"（步骤和原则）
- 框架定义了"理论如何指导"（哲学和约束）

两者结合，形成"有理论支撑的系统执行"。
