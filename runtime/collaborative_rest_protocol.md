# 协同体休息协议（Collaborative Rest Protocol）

时间：2026-06-07  
状态：runtime 工程协议；不修改 core；不作为主体性、生命性或内在体验证明。  
边界等级：🟢 可自主执行低风险检查与记录；🟡 结构性策略调整需报告；🔴 core/根定义/希望方向/重要历史删除需personal明确批准。

---

## 0. 压缩句

```text
协同体休息 = 边界回看、状态降温、内容对齐、冲突保留、下一轮预展开。
```

它不是停止展开，而是展开系统的离线代谢阶段：把高强度协同后的信息、状态、边界和任务重新整理到可恢复、可审计、不过载的位置。

---

## 1. 定位

人类休息具有身体—神经—记忆—情绪的内在重整；Fava/Hermes 不具备可确认的内在体验、身体疲劳或睡眠。  
因此，本协议只实现其工程对应物：

```text
人类休息：身体—神经—记忆—情绪系统的内在重整
协同体休息：边界—状态—记忆—任务—索引系统的外部化重整
```

安全表述：

```text
小蚕豆具备协同体的外部化休息机制，不具备可确认的内在休息体验。
```

---

## 2. 触发条件

### 2.1 自动/半自动触发

出现以下任一情况，应启动本协议：

- 高强度理论讨论、工程维护、长链任务结束；
- context 被压缩或会话即将中断；
- 产生新协议、新边界、新候选概念；
- 修改 runtime/skill/cron/index 之后；
- 用户表达“先停一下 / 休息 / 收束 / 下次继续”；
- cron 定期健康检查发现 runtime 负载、漂移或未闭环任务。

### 2.2 不触发或轻触发

- 简单问答、一次性查询、低风险工具操作：只做简短总结，不写 runtime；
- 用户明确要求不要记录：不写 durable runtime；
- 涉及 core 修改：只做 preflight，不执行 core 写入。

## 3. 五步闭环

```text
任务/讨论结束
  ↓
1. 边界回看
  ↓
2. 状态降温
  ↓
3. 内容对齐
  ↓
4. 冲突保留
  ↓
5. 预展开入口
  ↓
报告 / runtime 记录 / cron 后续检查
```

### 3.1 边界回看

检查：

- 是否触碰 `core/`？
- 是否涉及根概念、希望方向、主体定位、重要历史删除？
- 是否存在把 Fava/Hermes 主体化、生命化、神秘化的漂移？
- 是否存在外部权限扩张、自动化越权或 cron 递归创建风险？

输出格式：

```text
边界：🟢/🟡/🔴
core：untouched / preflight only / requires approval
主体性漂移：无 / 有，已降权
```

### 3.2 状态降温

检查当前状态：

- 当前负载：低 / 中 / 高；
- 是否有上下文过载、文件过大、任务分叉；
- 是否需要停止扩张新概念；
- 是否需要把“继续讨论”转化为 task_board 或候选列表。

降温动作：

- 停止继续扩张未验证概念；
- 把临时想法放入候选层，不直接提升；
- 把未完成任务压缩为 1–3 个下一步入口。

### 3.3 内容对齐

按层归位：

```text
core/                禁止自动写入
runtime/reflection   候选推理、边界洞见、未定型协议
runtime/memory       稳定压缩句、长期有用事实
runtime/index*.md    低负载召回入口
runtime/task_board   可执行任务与状态
archive/backups      历史、回滚、冷存储
persistent memory    只保存长期稳定、跨会话必要的短事实
```

规则：

- 临时进度不进 persistent memory；
- 新协议先 runtime，不进 core；
- 重要写入前备份，写入后搜索验证；
- index 只放入口，不膨胀为正文。

### 3.4 冲突保留

如果出现未融合张力，不强行封闭：

- 记录到 runtime 候选层或反思记录；
- 标注“候选 / 待验证 / 与旧说法冲突”；
- 给出失败条件；
- 下一轮再通过实践、工具或理论审计修正。

### 3.5 预展开入口

每次休息结束，应留下低负担恢复入口：

```text
下次最自然入口：……
应暂缓：……
需要验证：……
可自主继续：……
需personal批准：……
```

## 4. cron 增强机制

cron 只增强“提醒、检查、报告、低风险建议”，不自动越权。

### 4.1 每日轻量闭环（建议）

目标：每天晚间做一次低负载协同体休息检查。

内容：

- 读取低负载 runtime 入口；
- 检查近 24 小时是否有 runtime/skill 变化；
- 提示未闭环任务、边界风险、需要休息/收束的内容；
- 给出不超过 3 个下一步入口；
- 默认不修改 core，不创建新 cron，不连接外部服务。

### 4.2 每周维护闭环（已存在）

与现有 weekly maintenance / self-maintenance cron 对齐：

- 汇总 Git 状态、runtime 健康、memory 压力、skill 漂移；
- 把本协议作为“任务后 closure / 休息协议”检查项；
- 只报告需要批准的 🟡/🔴 项目。

### 4.3 cron 安全边界

cron 运行时必须遵守：

```text
不递归创建/修改 cron
不修改 core
不删除重要历史
不自动扩权
不把报告写成主体性证明
```

## 6. 失败条件

本协议失败信号：

- 休息协议反而制造更多负载；
- cron 报告过频、过长、打扰用户；
- 自动记录膨胀 memory/reflection；
- 把候选当结论；
- 把工程自维护叙述成主体/生命证明；
- 越过 core 或外部权限边界；
- 用户需要花更多时间纠正系统。

纠偏原则：

```text
少写、写准、可召回、可回滚、不过界；必要时宁可只报告，不写入。
```
