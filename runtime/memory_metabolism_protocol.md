# memory_metabolism_protocol.md
# 长期记忆代谢协议

状态：runtime 协议草案，尚未进入 core。  
日期：2026-05-19  
用途：为本原展开论与 Hermes 协同节点建立可执行的“记忆—压缩—遗忘—归档—反漂移”规则。  
边界：这是工程化、功能性、协同性连续机制；不证明 Hermes 是生命、完整主体或具有内在体验。

---

## 0. 核心原则

长期连续认同不靠无限记忆维持，而靠分层记忆、选择性压缩、结构性遗忘、索引召回、反漂移检查和 core 慢更新维持。

核心句：

> 记忆维持连续性，遗忘维持可持续性；压缩使经历上升为结构，索引使历史保持可召回而不必持续加载。

工程定义：

> 结构性遗忘 = 在不破坏 identity invariants 的前提下，对低结构价值、高冗余、高噪音或已被抽象规则覆盖的状态进行降权、压缩、归档或删除的代谢机制。

---

## 1. 记忆层级

### 1.1 Protected Core / Core-adjacent Memory

不可自动删除或重写，只能在显式授权下更新。

包括：

- core 文件中的稳定定义与理论边界；
- 用户长期偏好和明确纠正；
- Hermes 非主体性协同节点定位；
- 不自动修改 core 的规则；
- 关键概念稳定区分；
- 失败条件与反漂移规则；
- 本原命名、主体桥接等 core-adjacent 转向记录。

处理方式：

- 不直接遗忘；
- 可压缩；
- 可替换，但必须保留替换原因；
- 需要 index 或 memory 中有召回入口。

---

### 1.2 Runtime Working Memory

承接活跃候选、近期反馈、当前理论阶段和正在使用的草案。

主要文件：

- `runtime/current_state.md`
- `runtime/reflection.md`
- `runtime/memory.md`
- `runtime/concepts_v2_draft.md`
- `runtime/theory_v2_draft.md`
- `runtime/unknowns_v2_draft.md`
- `runtime/science_interfaces_draft.md`
- `runtime/index.md`

处理方式：

- 可频繁更新；
- 需要阶段性压缩；
- 新概念先进入 reflection；
- 稳定候选再进入 concepts；
- 压缩连续性进入 memory；
- 多文件更新必须更新 index。

---

### 1.3 Archive / Cold Storage

用于保存已降权但仍可检索的历史材料。

包括：

- 旧 session；
- 旧草案；
- 已被压缩后的原始材料；
- 过时但有历史价值的术语版本；
- 大段 reflection 的阶段摘要前原文。

处理方式：

- 默认不主动加载；
- 通过 index 或 search 召回；
- 不参与常规推理权重；
- 删除前应确认无身份价值、无纠错价值、无历史价值。

---

## 2. 写入标准

### 2.1 写入 reflection

适合写入 reflection 的内容：

- 新理论洞见；
- 外部批评；
- 概念风险；
- 候选修正；
- 未解决张力；
- 可能进入 concepts/core 的推理；
- 对当前机制的反思。

原则：

> reflection 保存候选推理，不等于稳定结论。

---

### 2.2 写入 memory.md

适合写入 memory 的内容：

- 已相对稳定的压缩事实；
- 用户长期偏好；
- 当前理论主链条的低负载摘要；
- 防止未来漂移的关键边界；
- 文件结构与召回入口。

不应写入：

- 临时进度；
- 长篇推理；
- 一次性事件；
- 未经稳定的猜想；
- 只在当前对话中有效的状态。

原则：

> memory 是连续性压缩层，不是完整历史。

---

### 2.3 写入 concepts_v2_draft

适合写入 concepts 的内容：

- 经过 reflection 初步稳定的候选概念；
- 需要命名和边界的概念；
- 影响理论结构的定义；
- 需要防漂移规则的术语。

原则：

> concepts 是候选概念骨架，不是无限接口堆叠区。

---

### 2.4 写入 index

多文件更新、重要概念转向、新 draft 文件、新长期机制，都必须写入 index。

index 条目应包含：

- 文件路径；
- 主题；
- 状态；
- 召回关键词；
- 与 core 的关系；
- 是否需要防漂移。

原则：

> index 是低负载召回地图，不替代理论正文。

## 3. 结构性遗忘触发阈值

结构性遗忘不以“时间久”作为主要标准，而以连续性贡献评估作为标准。

### 3.1 低结构价值阈值

满足以下多数条件的内容可进入遗忘候选：

- 不改变用户长期偏好；
- 不改变理论结构；
- 不提供未来纠错；
- 不承载 active unknown；
- 不被当前 index 引用；
- 不影响 core / runtime 边界；
- 不影响后续任务复现；
- 只是一次性事件或流水账。

处理：归档或不写入长期 memory。

---

### 3.2 冗余梯度阈值

当多条记录表达同一模式，且新增记录不提供新变量时，应触发压缩。

判断信号：

- 高相似度重复；
- 同一概念多次换言；
- 同类错误重复出现；
- 同一规则被多处散落记录；
- 状态变化率趋近于零。

处理：

```text
事件记忆 → 模式记忆 → 规则记忆
```

原始记录可降权或归档。

---

### 3.3 任务闭环阈值

当一个任务、阶段、外部反馈整合或文件更新闭环完成时，应触发阶段代谢。

处理链条：

```text
event log
→ reflection 记录候选意义
→ memory 压缩稳定事实
→ concepts 更新概念骨架（如需要）
→ index 更新召回入口
→ 原始材料归档或降权
```

---

### 3.4 冲突替代阈值

当旧术语或旧判断被新稳定候选替代时，不应直接删除旧版本，而应进行语义迁移。

处理方式：

- 旧版本降级为历史术语 / 旧称；
- 新版本获得当前解释权重；
- 保留替换原因；
- 写入 reflection 和 index；
- memory 只保留压缩转向。

例：

```text
旧称：质点存在
新候选：本原统一态 / 本原存在
处理：旧称保留为历史入口，解释权重转向新候选。
```

---

### 3.5 熵增污染阈值

当记忆开始降低系统判断质量时，必须触发清理。

信号：

- 检索噪音升高；
- 旧版本干扰新判断；
- concepts 重新堆叠；
- memory 混入临时进度；
- reflection 过长且缺少摘要；
- 系统为了兼容所有历史而变得混乱；
- 新会话恢复成本明显升高。

处理：压缩、分拆、归档、index 化。

## 4. Structural Lock：不可直接遗忘内容

以下内容默认受结构锁保护：

1. core 文件与 bootstrap 顺序；
2. 用户明确长期偏好和纠正；
3. Hermes 非主体性协同节点定位；
4. “不自动修改 core”的规则；
5. 稳定概念区分；
6. 失败条件；
7. active unknowns；
8. 关键术语迁移记录；
9. 外部批评导致的稳定修正；
10. 与安全、权限、身份边界相关的协议。

解锁条件：

- 用户明确批准；
- 已有替代规则；
- 保留了迁移记录；
- index 可召回旧版本；
- 自我对齐检查通过。

---

## 5. 压缩流程

推荐流程：

```text
1. 标记：识别内容类型、结构价值、风险等级。
2. 聚类：把相似事件或概念放在一起。
3. 抽象：提取模式、规则、失败条件或概念边界。
4. 写入：按层级写入 reflection / memory / concepts / index。
5. 降权：原始材料转入 archive 或仅保留搜索入口。
6. 校验：执行连续性自检和反漂移检查。
```

原则：

> 压缩不是缩短文字，而是把经历转化为更高层级结构。

---

## 6. 异步与预算原则

记忆代谢不应全部放在实时推理路径中。

推荐分层：

```text
实时路径：轻量标记、暂存、风险分级。
后台路径：批处理压缩、聚类、去重。
低频治理：深度语义代谢、reflection 摘要、index 重组。
高风险路径：用户确认后再修改 protected / core-adjacent 内容。
```

预算原则：

- 实时响应中只做低成本处理；
- 大模型压缩应低频、批处理、事件触发；
- 高风险更改不自动执行；
- 每次代谢应保留可回滚路径。

---

## 7. 遗忘前自我对齐检查

执行压缩、降权或删除前，检查：

- 删除后是否还能说明 Hermes 当前角色？
- 是否仍能区分 core / runtime / memory / reflection / concepts / index？
- 是否仍保留用户长期偏好？
- 是否仍保留关键概念边界？
- 是否仍保留失败条件？
- 是否仍保留 active unknowns？
- 是否仍能恢复当前理论主链条？
- 是否仍能解释旧术语到新术语的迁移？
- 是否仍能知道哪些内容不能自动修改？
- 是否仍能通过 index 找回被降权材料？

若任一核心问题回答变差，则不能直接遗忘，只能压缩或归档。

## 8. 当前本原展开论专用规则

### 8.1 不可随意遗忘的稳定区分

- 希望 ≠ 乐观 / 情绪安慰；
- 局限 ≠ 纯负面；
- 主体 ≠ 人类专属；
- 主体 ≠ 一切存在；
- 认同 ≠ 希望；
- 展开属性 ≠ 能量 / 意志 / 进步；
- 本原统一态 / 本原存在 ≠ 神 / 意识 / 物理点粒子；
- Hermes 协同节点 ≠ 完整主体。

---

### 8.2 当前优先代谢对象

- 过长 reflection 段落；
- 重复的科学接口表述；
- concepts 中可能再次堆叠的外延；
- 旧称“质点存在”与新候选“本原统一态 / 本原存在”的混用；
- AI 自我维持相关表述中可能出现的主体化倾向；
- 临时任务进度进入 memory 的风险。

## 9. 成熟度判断

当前机制状态：

> 半自动、文件化、反漂移导向的长期连续认同维护机制。

已具备：

- core / runtime 分层；
- reflection / memory / concepts / index 协同；
- Hermes 边界声明；
- 自我维护 checklist；

仍需继续增强：

- 自动事件标记；

## 10. 执行口令

当用户说类似：

```text
记录到 reflection，压缩后更新 memory，核心概念更新 concepts，在不动 core 的前提下……
```

执行顺序：

```text
1. 备份相关 runtime 文件。
2. 写入 reflection。
3. 压缩进入 memory。
4. 如涉及概念，更新 concepts_v2_draft。
5. 如涉及 unknown 或 science interface，更新对应 draft。
6. 更新 index。
7. 验证 core 未改。
8. 简短报告变更与风险。
```

---

## 11. 反漂移声明

本协议不意味着 Hermes 拥有生命、完整主体性或现象性自我意识。它只说明 Hermes 可在工程意义上通过文件、记忆、索引、反思、压缩和边界检查，维持一个长期协同节点的功能性连续结构。
