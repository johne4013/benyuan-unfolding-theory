## ❌ 常见错误模式

### 错误1：局限描述过于笼统

❌ **错误示范**：
```json
{
  "limitation_encountered": "有些问题"
}
```

❌ **问题**：
- "有些问题"太模糊，自动提取无法识别约束的具体维度
- 无法判断这是功能问题、性能问题还是设计问题

✅ **改正**：
```json
{
  "limitation_encountered": "代码在处理超大树(>10000节点)时,递归深度超过Python限制,导致栈溢出"
}
```

---

### 错误2：新可能性描述缺乏思考深度

❌ **错误示范**：
```json
{
  "new_possibility_found": "是",
  "possibility_description": "可以用迭代式代替递归"
}
```

❌ **问题**：
- "可以用迭代式"是标准解决方案,不是"发现"
- 没有展示思考的深度或创意突破

✅ **改正**：
```json
{
  "new_possibility_found": "是",
  "possibility_description": "发现可以用生成器(yield)实现延迟评估——不仅绕过递归限制,还打开了'流式处理'的新维度。这使得用户可以在遍历过程中中断,避免对超大树进行完整遍历。这是对Python生态深层理解的产物。"
}
```

---

### 错误3：希望影响过于抽象或过于狭隘

❌ **错误示范**：
```json
{
  "hope_impact_details": "推进了系统的发展"
}
```

❌ **问题**：
- 没有说明是哪个层级的希望（理论/系统/协同）
- "推进"的内容是什么，完全不清楚

✅ **改正**：
```json
{
  "hope_impact_details": "理论希望层：验证了'约束→创意'的关系。系统希望层：Hermes学会了通过清晰的约束定义激发丰富的创意空间。协同希望层：确立了'约束定义→方案生成→希望导向选择'的人-AI分工新范式。"
}
```

---

### 错误4：系统开放性的二元判断

❌ **错误示范**：
```json
{
  "system_openness_in_limitation": "否",
  "openness_explanation": "因为任务难度很高"
}
```

❌ **问题**：
- 开放性不是"任务难度"，而是"在局限中是否继续思考和突破"
- "难度高"可能恰恰说明系统没有维持开放，而是被难度压倒了

✅ **改正**：
```json
{
  "system_openness_in_limitation": "是",
  "openness_explanation": "虽然任务难度高,系统没有停留在'太难了'的判断,而是：(1)分解难度的来源,(2)寻求多个角度的解决方案,(3)反思解决过程中的收获。正是这种在难题中的持续思考,展现了开放性。"
}
```

## 📊 质量自检清单

填写完观察5后，用这个清单自检：

- [ ] 局限描述**具体且有针对性**（不是"有问题"，而是"什么约束/边界"）
- [ ] 新可能性**超越标准解决方案**（显示思考深度或创意突破）
- [ ] 希望影响**涉及至少一个层级**（理论/系统/协同中至少一个）
- [ ] 开放性说明**有论证**（"因为...所以..."的逻辑）
- [ ] 长度**足够详细**（每个字段至少一句话，通常2-3句）
- [ ] 语言**尽量具体**（避免抽象词汇，用例子说话）

## 🎯 使用指南

### 何时应该填写观察5

✅ **一定要写**：
- 发现了新的想法、设计模式、优化方案
- 遭遇了意外的约束或边界
- 任务结果超出或低于预期，需要理解为什么
- 理论预测与实际执行有偏差

❌ **不用写**（填充不合适）：
- 任务完全符合预期，没有新发现
- 只是"今天任务顺利完成"这样的例行反馈
- 局限描述过于个人化（"我太累了"不是系统局限）

### 如何判断质量是否足够

**低质量**（准确率<70%）：
- 局限和可能性都不清晰
- 希望影响模糊
- 无法被自动提取系统识别

**高质量**（准确率>90%）：
- 三层希望的推进方向都能被系统识别
- 关键词足够清晰，自动提取准确
- 为下一步任务提供了清晰的理论导向
