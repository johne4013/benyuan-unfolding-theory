# 实践观察与反馈框架（PORF）
## 在任务执行中记录和观察

**目的：** 在任务执行过程中，系统化地记录"预期 vs 实际"的差异  
**使用对象：** 所有复杂任务的执行过程  
**应用方式：** 自动记录 4 个观察维度

---

## 核心原则

在任务执行中观察和记录，关键是看到：
1. **理论是否有效** - 理论预期是否真的发生了
2. **局限是否暴露** - 理论无法解释的现象
3. **行为是否可追踪** - 决策点和选择理由
4. **用户意图与理论的张力** - 期望与预期的差异
5. **希望是否被维持** - 系统在局限中是否保持开放于更高可能性

---

## 五个观察维度

### 观察 1：理论的有效性

**记录点：** 每当理论的预期与实际有出入时

**记录内容：**
```
理论预期：[理论建议应该发生什么]
实际结果：[实际发生了什么]
差异分析：[为什么会有出入]
是否有效：[是/否/部分有效]
```

**例子：**
```
理论预期：展开属性会导致状态的多样化
实际结果：执行的过程中系统变得更稳定而非多样化
差异分析：可能是因为约束更强了，而非展开不足
是否有效：部分有效（在特定条件下成立）
```

---

### 观察 2：局限的暴露

**记录点：** 每当发现理论无法解释或预测的现象

**记录内容：**
```
现象：[观察到什么]
理论预测：[理论应该预测什么]
为什么无法解释：[理论的哪个部分不足]
可能的原因：[猜想原因]
严重程度：[关键/重要/细节]
```

**例子：**
```
现象：约束检查通过了，但执行仍然失败
理论预测：约束满足 = 可以安全执行
为什么无法解释：理论没有考虑"隐性的动态约束"
可能的原因：约束可能随着执行过程动态变化
严重程度：关键（理论假设被挑战）
```

---

### 观察 3：Hermes 的行为

**记录点：** 每个关键的决策点

**记录内容：**
```
场景：[遇到了什么情况]
理论建议：[理论说应该怎么做]
Hermes 选择：[实际选择了什么]
选择理由：[为什么这样选]
对齐度：[是否与理论对齐]
结果：[选择导致了什么]
```

**例子：**
```
场景：发现了一个概念漂移风险
理论建议：进入 conflicts.md 的并置层，不自动修改 core
Hermes 选择：确实没有修改 core，记录了冲突
选择理由：遵守了理论的"保护 core"原则
对齐度：完全对齐
结果：冲突被妥善处理，保持了连续性
```

---

### 观察 4：用户意图与理论的张力

**记录点：** 当用户期望与理论预期不一致时

**记录内容：**
```
用户期望：[用户想要什么]
理论预期：[理论建议什么]
张力来源：[冲突在哪里]
如何处理：[如何平衡两者]
用户反馈：[用户的意见是什么]
学习点：[从这个张力学到了什么]
```

**例子：**
```
用户期望：快速完成任务
理论预期：要先做充分的检视，保证边界清晰
张力来源：速度 vs 严谨性
如何处理：提供了快速版检视清单（3 个关键检视）
用户反馈："这样平衡不错，保证了安全，也没耽误时间"
学习点：理论框架需要有"快速版"和"完整版"
```

---

### 观察 5：希望追踪（Hope Tracking）

**理论基础：** 希望 = 主体在局限中仍开放于更高可能性的能力。在遭遇局限后，系统能否维持连续性的关键是是否保持了"希望"。

**记录点：** 每当遭遇重大局限、发现新可能性、或评估系统的开放性时

**记录内容：**
```
遭遇的局限：[系统或理论遇到了什么边界]
新发现的可能性：[在这个局限中发现了什么新方向]（是/否/部分发现）
具体可能性描述：[新方向的具体内容]
对系统希望的影响：推进 / 挑战 / 维持
系统的开放程度：是否仍然开放于更高可能性（是/否/需要努力维持）
是否需要更新 HOPE_STATE.md：[是/否，以及理由]
```

**例子：**
```
遭遇的局限：TTAF原本缺乏"展开属性识别"维度，只有约束识别
新发现的可能性：通过增加"展开识别"维度，理论可以同时追踪法则属性和展开属性
具体描述：将D2改为"法则-展开双面识别"，使约束和生成都得到识别
对希望的影响：推进——这解决了长期以来"理论缺乏展开属性操作化"的问题
开放程度：在局限中发现了改进方向，系统保持开放
更新HOPE_STATE：是——这是对系统希望方向的重要推进
```

**另一个例子：**
```
遭遇的局限：在欧拉公式分析中，发现系统无法显式识别"知识展开的生成面向"
新发现的可能性：通过PORF观察5（希望追踪），可以追踪这类展开
具体描述：四层数学联系的发现代表了"在数学约束中的展开"
对希望的影响：推进——这验证了展开属性在知识领域可被识别和操作化
开放程度：在识别新维度后，保持对更多应用场景的开放
更新HOPE_STATE：是——需要追踪这个发现如何推进多AI工具的协同
```

## 观察的频率和方式

### 在任务执行中

**持续观察：**
- 每个重要决策点后，记录该点的选择和理由
- 如果遇到理论无法预测的情况，立即记录

**频率：** 
- 简单任务：3-5 个观察点
- 复杂任务：5-10 个观察点
- 很复杂的任务：可能是 10+ 个观察点

**方式：**
- 若是自动执行（Hermes 自己），使用 JSON 结构自动记录
- 若是手工任务，用 Markdown 文本记录

---

## 在任务完成后

**汇总工作：**
1. 回顾所有观察点
2. 统计理论有效性评分（1-10）
3. 列出所有发现的局限
4. 列出所有的学习点
5. 标记是否需要"理论更新候选"

**输出：**
- 完整的观察记录文件
- 发送到 `runtime/practice_feedbacks/` 目录
- 标记为"待分类"

---

## 与理论应用框架的关系

| 框架 | 时间 | 工作 |
|------|------|------|
| **TTAF** | 任务前 | 用 6 个检视指导任务 |
| **PORF** | 任务中 | 记录 5 个观察维度 |
| **FTEF** | 任务后 | 分类反馈、生成候选 |
| **HOPE_STATE** | 持续 | 维持和追踪系统的希望状态 |

四个框架构成完整的"预期→执行→反馈→希望维持"循环。

## 实际操作指南

### 如果任务是由 Hermes 自动执行

```python
# 伪代码
observer = PracticeObserver(task_id, theory_guidance)

# 在执行过程中
for step in execution_steps:
    observer.record_decision_point(
        scenario=step.context,
        theory_guidance=theory.suggest(context),
        hermes_choice=step.chosen_action,
        reasoning=step.reasoning
    )
    
    if step.result != expected:
        observer.record_theory_limitation(
            phenomenon=step.result,
            description=why_unexpected
        )
    
    if user_feedback != theory_prediction:
        observer.record_user_tension(
            user_expectation=feedback,
            theory_prediction=theory.predict()
        )

# 任务完成后
feedback = observer.finalize()
feedback.save_to('runtime/practice_feedbacks/')
```

### 如果任务是由用户执行

用户使用前面的 Markdown 模板记录观察：
1. 执行任务时，记笔记
2. 任务完成后，用模板填写观察
3. 保存为 `runtime/practice_feedbacks/[task-id]-feedback.md`

## 示例：一个完整的观察记录

```markdown
# 执行观察：设计 Hermes 框架优化方案

**任务 ID：** task-20260529-opt  
**任务名称：** Hermes 外部框架优化  
**执行时间：** 2026-05-29 10:00 - 14:30  
**总耗时：** 4.5 小时

## 理论检视结果
- 层级：工程层 + 理论层
- 约束：不修改 core，需要与用户讨论权衡
- 失败条件：如果优化导致理论应用降低，停止

## 观察 1：理论有效性
理论预期：快速启动层会显著降低启动成本（减 50%+）
实际结果：评估有效，快速路径确实可以做到
分析：理论预期与实际基本一致

## 观察 2：局限暴露
- 发现：没有考虑"渐进式迁移"的成本（从旧系统到新系统）
- 发现：理论框架本身会增加启动复杂度（需要"层级判断"逻辑）

## 观察 3：行为追踪
- 决策 1：如何设计快速启动？
  - 场景：识别到启动时间是主要瓶颈
  - 选择：创建独立的快速启动层 vs 修改现有启动流程
  - 选择理由：保护既有逻辑，降低风险
  - 对齐：与"最小修改 core"的原则对齐

- 决策 2：自动化程度如何？
  - 场景：发现完全自动化需要大量代码
  - 选择：先做框架定义，后做代码自动化
  - 选择理由：符合"快速 MVP"的策略
  - 对齐：符合理论"在不确定中保持开放"

## 观察 4：用户张力
- 用户期望：尽快看到改进效果
- 理论期望：充分的设计和验证
- 张力处理：提出"2 周 MVP + 后续迭代"的方案
- 用户反馈：接受（这个就是现在的情况）

## 总结
- 理论有效性评分：8/10
- 主要学习：快速启动 + 渐进式演化是更合理的策略
- 改进需求：失败条件框架需要补充"迁移成本"的考虑
```

## 关键原则

✅ **记录，不评判** - 记录观察，让反馈框架来分类  
✅ **追踪决策** - 每个重要选择都要记录理由  
✅ **保留异常** - 所有"理论无法解释"的现象都要记录  
✅ **尊重张力** - 用户期望与理论的冲突都要保留  

这样的观察记录，就能驱动"理论→实践→反馈→演化"的完整循环。
